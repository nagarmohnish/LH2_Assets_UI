01 · User Flows

Three flows cover the lifecycle.

New subscription, duplicate attempt, and magic-link activation. The monthly flow mirrors the lifetime architecture — the difference is Stripe Subscriptions (not PaymentIntents) and a recurring-billing cancellation path.

A

New subscription

First-time visitor subscribes $1/mo
Start demo
User sees sticky banner on home/article page — clicks CTA
Lands on /subscribe — enters email
Frontend calls /sub/init with email
Backend checks wp_stb_subscribers — email not found — creates Stripe Customer + Subscription (status: incomplete) — returns clientSecret for the first invoice's PaymentIntent
Stripe Elements mount — user pays $1 via card, Link, Apple/Google Pay
Stripe redirects to /sub-success
Success page auto-fires /sub/send-login — magic link emailed
Stripe webhook (invoice.paid, customer.subscription.updated) upserts row into wp_stb_subscribers with status=active and current_period_end
B

Duplicate / cancelled attempt

Already-subscribed or previously-cancelled email
See blocked state
Returning user enters email on /subscribe
/sub/init finds email — checks subscription status:
If status=active — returns blocked: true, frontend shows “Already subscribed — send login link”
If status=canceled or past_due — returns resubscribe: true, frontend reactivates the Stripe subscription instead of creating a new one
User clicks login link — magic link emailed via /sub/send-login
C

Magic link activation

First-time or returning login
See activation
User clicks link in email — hits /sub-login?stb_magic={token}
Backend validates token — creates WP user if not exists
Links entitlement row to user (wp_stb_subscribers.user_id = user.ID)
Sets WP auth cookie — redirects to home
Ad-hiding gate checks status=active AND current_period_end > now — hides ads

02 · Data Model

One table. Recurring-aware.

wp_stb_subscribers stores subscription state. Unlike the lifetime table, it tracks a rolling current_period_end and a status that webhooks update on each invoice.

03 · API Endpoints

Four routes, plus billing portal.

Three REST endpoints for the flow, one frontend route for magic-link activation, and a Stripe Customer Portal redirect for self-serve cancellation / card updates.

/wp-json/stb/v1/sub/initPOSTEmail gate + Customer/Subscription creation or resume
/wp-json/stb/v1/sub/send-loginPOSTGenerate + email magic-link token

04 · Security & Resilience

The gotchas already handled.

Failed invoices, card expiry, cancellation grace period, webhook ordering, refunds.

Security

  • Webhook signature verification on every Stripe event
  • IP rate limiting on /sub/init (10 req/min)
  • Silent response on /sub/send-login for unknown emails
  • Magic-link tokens — 48-char random, SHA-256 hashed, single-use, 30-min expiry
  • Idempotent webhook handling — upsert by stripe_subscription_id
  • Billing portal redirects are signed short-lived Stripe sessions

Failure Modes

  • Invoice payment fails — status becomes past_due, Stripe auto-retries via Smart Retries; gate still returns true for a 3-day grace window
  • User cancels — Stripe sets cancel_at_period_end=true; gate remains true until current_period_end, then flips to false on subscription.deleted webhook
  • Card expires — Stripe emails the user + sends invoice.payment_failed; site shows a “update card” banner on next login
  • Webhook out of order — always check current_period_end against Stripe API, not just cached value
  • Refund issued — charge.refunded flips status to canceled immediately
