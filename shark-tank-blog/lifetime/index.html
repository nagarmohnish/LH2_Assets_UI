01 · User Flows

Three flows cover the lifecycle.

New purchase, duplicate purchase, and magic-link activation. Each flow has a corresponding demo page you can click through below.

A

New purchase

First-time visitor buys lifetime access
Start demo
User sees sticky banner on home/article page — clicks CTA
Lands on /go-ad-free — enters email
Frontend calls /adfree/init with email
Backend checks wp_stb_adfree — email not found — creates Stripe PaymentIntent — returns clientSecret
Stripe Elements mount — user pays via Apple Pay / Google Pay / Link / card
Stripe redirects to /ad-free-success
Success page auto-fires /adfree/send-login — magic link emailed
Stripe webhook (payment_intent.succeeded) inserts row into wp_stb_adfree (no WP user yet)
B

Duplicate purchase attempt

Paid user, not logged in, tries again
See blocked state
Returning paid user (not logged in) enters email on /go-ad-free
/adfree/init finds email in wp_stb_adfree — returns blocked: true
Frontend replaces checkout UI with “Already purchased — send login link” state
User clicks — /adfree/send-login — magic link emailed
C

Magic link activation

First-time or returning login
See activation
User clicks link in email — hits /adfree-login?stb_magic={token}
Backend validates token from transient — creates WP user if not exists
Links entitlement row to user (wp_stb_adfree.user_id = user.ID)
Sets WP auth cookie — redirects to home
Ad-hiding gate now returns true for this session and all future logged-in visits

02 · Data Model

One table, one source of truth.

wp_stb_adfree holds entitlement. A WordPress user is created only on first magic-link click — the table is canonical, the user is just the login vehicle.

04 · Security & Resilience

The gotchas already handled.

Email enumeration, duplicate payments, lost magic links, refund reversal, webhook retries.

Security

  • Webhook signature verification on every Stripe event
  • Server-side PI verification on /ad-free-success — never trust the URL param alone
  • IP rate limiting on /adfree/init (10 req/min) — mitigates email enumeration
  • Silent response on /adfree/send-login for unknown emails — prevents enumeration
  • Magic-link tokens — 48-char random, SHA-256 hashed in storage, single-use, 30-min expiry
  • Idempotent webhook inserts — unique constraint on payment_intent_id + INSERT IGNORE

Failure Modes

  • Webhook delayed past success page load — page shows generic “check your email”; magic link still arrives when webhook fires
  • User loses magic link — “Send login link” flow on /go-ad-free after email-blocked state
  • Refund issued — charge.refunded sets refunded_at; entitlement gate returns false
  • Duplicate webhook delivery — unique constraint makes insert idempotent
  • Expired magic link — user re-requests from /go-ad-free duplicate-block flow
05 · Entitlement States

Who sees what.

The ad-hiding gate is a single function: logged in AND entitlement exists on user — hide ads. Otherwise show ads.

State Logged in? Entitlement? Sees ads? Banner?
New visitorNoNoYesYes
Paid, not activatedNoYes (by email)YesYes
ActivatedYesYes (linked)NoNo
RefundedAnyRevokedYesYes

The paid-but-not-activated state is the critical drop-off. Magic-link auto-send on /ad-free-success is the primary mitigation; a 24-hour nudge email for unlinked entitlements is recommended follow-up work.
